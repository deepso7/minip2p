Small enough to understand. Complete enough to connect.

`minip2p` gives applications one synchronous [`Endpoint`](/guides/mental-model#endpoint)
for identity, QUIC connections, built-in Identify and Ping, and application
protocol streams. Optional features add NAT traversal, pubsub, and peer
discovery without changing the base API.

```rust
use minip2p::Endpoint;

fn main() -> Result<(), minip2p::Error> {
    let mut node = Endpoint::builder()
        .agent_version("my-app/0.1.0")
        .bind_quic("127.0.0.1:0")?;

    println!("listen={}", node.listen()?);

    Ok(())
}
```

## The shape of minip2p

- **QUIC only.** One transport, with mutual libp2p TLS authentication built in.
- **No async runtime.** Your application calls `poll`, `next_event`, or a
  focused wait when it wants the endpoint to make progress.
- **Pre-1.0.** Breaking API changes are possible, and the crates are not yet
  published to crates.io.
- **libp2p wire formats.** Identify, Ping, Circuit Relay v2 client flows,
  AutoNAT probes, DCUtR, gossipsub, floodsub, and signed discovery speak the
  protocol wire formats those stacks expect.
- **Sans-I/O underneath.** Protocol logic remains deterministic; sockets,
  clocks, and blocking waits live in the adapters.

:::note
These docs begin with the app-facing `minip2p::Endpoint`. Lower-level
Sans-I/O crates remain available, but their narrative guides are planned for a
later documentation pass.
:::

## Current constraints

| Constraint | What it means |
| --- | --- |
| Rust 1.88+ | This is the workspace's minimum supported Rust version. |
| QUIC transport | TCP, WebSocket, and WebTransport adapters are out of scope. |
| Ed25519 identity | Endpoint identities and signed application messages use Ed25519. |
| Caller-driven | There is no executor or hidden async task driving the node. |
| Relay client only | Bring a Circuit Relay v2 server when a relayed path is required. |
